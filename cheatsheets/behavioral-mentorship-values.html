STORY A Entity-resolution defect — failure / accuracy / accountabilityДефект резолюции сущностей — провал / точность / ответственность
  • S — At Meta, abuse-detection metrics drifted. Root cause: an entity-resolution defect mis-merging/splitting identities across 6.2B records (34% of messaging threads), silently corrupting 100+ downstream metrics & ML features.
  • T — Own root cause, fix, backfill, and rebuilding trust in the numbers without breaking 100+ consumers.
  • A — Reproduced on a slice → isolated the join/dedup logic → validated corrected logic vs. ground-truth before touching prod → coordinated backfill → added automated validation checks so the same defect class is caught by the pipeline, not by a human noticing drift weeks later.
  • R — Restored accuracy for 100+ metrics/features; turned a silent-corruption mode into a loud, early one.
  • S — В Meta метрики детектирования абьюза поползли вниз. Корневая причина: дефект резолюции сущностей, ошибочно объединявший/разделявший идентичности на 6.2 млрд записей (34% мессенджерных тредов), молча ломая 100+ нижестоящих метрик и ML-фич.
  • T — Найти корневую причину, исправить, забэкфиллить, восстановить доверие к числам без поломки 100+ консьюмеров.
  • A — Воспроизвёл на срезе → изолировал логику join/dedup → проверил исправленную логику на ground-truth до трогания прода → координировал backfill → добавил автоматические проверки валидации, чтобы тот же класс дефектов ловился пайплайном, а не человеком, заметившим дрифт через недели.
  • R — Восстановил точность для 100+ метрик/фич; превратил режим молчаливой порчи в громкий и ранний.
The line that wins "The bug wasn't the failure. The failure was that nothing in the system caught it. So the real fix was making correctness observable."
Выигрышная фраза «Баг — не провал. Провал в том, что ничто в системе его не поймало. Поэтому реальный фикс — сделать корректность наблюдаемой.»

Best for: failure/hard bug, accuracy & accountability, debugging depth, "data is wrong — now what."

Подходит для: провал/сложный баг, точность и ответственность, глубина дебага, «данные ошибочны — что дальше».

STORY B Unified data model / 10 logging systems — ownership & designЕдиная модель данных / 10 систем логирования — владение и дизайн
  • S/T — 10 fragmented logging systems, ~3.6B events/day, blocking cross-surface integrity analytics at billions-user scale.
  • A — Architected a unified data model consolidating all 10; eliminated 47% redundant data volume; unblocked cross-surface analytics.
  • R — One coherent model, big cost + clarity win, analytics unblocked.
  • S/T — 10 фрагментированных систем логирования, ~3.6 млрд событий/день, блокировали кросс-платформенную аналитику целостности на масштабе миллиардов пользователей.
  • A — Спроектировал единую модель данных, консолидировавшую все 10; устранил 47% избыточного объёма данных; разблокировал кросс-платформенную аналитику.
  • R — Одна когерентная модель, большой выигрыш в стоимости и ясности, аналитика разблокирована.

Best for: end-to-end ownership, design thinking, data modelling, modernization, scalability.

Подходит для: сквозное владение, дизайн-мышление, моделирование данных, модернизация, масштабируемость.

STORY C Triple migration at McMakler — ownership / tech debt / under pressureТройная миграция в McMakler — владение / технический долг / под давлением
  • S/T — Three concurrent platform problems, owned as tech lead.
  • A — GCP-native → Snowflake; proprietary orchestration → Airflow (killed 100s of K €/yr licensing); Postgres → dbt analytical-layer rebuild. De-risked via parallel run + row-for-row validation + per-domain cutover so reporting never broke. Added a Salesforce sync layer via Airflow.
  • R — €1M+ saved; modern stack; self-service analytics for 200+ users across 8 departments.
  • S/T — Три одновременные проблемы платформы, которые я владел как тех. лид.
  • A — GCP-native → Snowflake; проприетарная оркестрация → Airflow (убил сотни тысяч €/год лицензий); Postgres → переписал аналитический слой на dbt. Снизил риски через параллельный запуск + поэлементную валидацию + постепенный cutover по доменам, поэтому отчётность никогда не ломалась. Добавил слой синхронизации Salesforce через Airflow.
  • R — €1M+ сэкономлено; современный стек; self-service аналитика для 200+ пользователей из 8 отделов.
Senior tell Parallel-run + per-domain cutover + row-level validation = how you migrate critical data with no trust outage.
Маркер Senior Параллельный запуск + постепенный cutover по доменам + поэлементная валидация = как мигрировать критические данные без разрыва доверия.

Best for: end-to-end ownership, migrations, reducing tech debt, delivering under pressure.

Подходит для: сквозное владение, миграции, погашение технического долга, доставка под давлением.

STORY D Growing 2→7 & dbt code review — mentorship / raising the barРост команды 2→7 и ревью dbt-кода — менторство / повышение планки
  • S/T — Inherited a 2-person team, no standards, untested SQL feeding 200+ users (McMakler); later led 8 engineers across 8 brand domains, same disease at scale (Career.io).
  • A — Institutionalized dbt code review (checklist: tests present, naming, no SELECT *, incremental logic) + SQL optimization standards (partition pruning, materialization, no accidental cross-joins). Didn't just write rules — pair-reviewed early PRs, wrote exemplar models, ran "why this is faster" sessions. Gave each engineer an owned domain, reviewed toward independence.
  • R — Grew team 2→7, promoted 3 Junior→Mid/Senior; defects caught pre-prod; onboarding faster.
  • S/T — Унаследовал команду из 2 человек, без стандартов, непротестированный SQL кормил 200+ пользователей (McMakler); позже вёл 8 инженеров по 8 брендам, та же болезнь на масштабе (Career.io).
  • A — Институционализировал ревью dbt-кода (чеклист: наличие тестов, нейминг, нет SELECT *, инкрементальная логика) + стандарты оптимизации SQL (прунинг партиций, материализация, нет случайных cross-join). Не просто писал правила — парно ревьюил первые PR, писал образцовые модели, проводил сессии «почему это быстрее». Дал каждому инженеру домен, который он владеет, ревьюил к самостоятельности.
  • R — Вырастил команду 2→7, повысил 3 Junior→Mid/Senior; дефекты ловились до прода; онбординг ускорился.
The line that wins "My measure of mentorship is whether the person can pass their own review six months later. Promotions are the lagging indicator."
Выигрышная фраза «Мой показатель менторства — сможет ли человек через полгода сам пройти своё собственное ревью. Повышения — запаздывающий индикатор.»

Best for: mentorship, raising standards, conflict/pushback, influence.

Подходит для: менторство, повышение стандартов, конфликт/сопротивление, влияние.

  • Lead with scale/stakes in S so the interviewer knows it's a hard problem (6.2B records, 100+ consumers, €1M+, 8 brands).
  • Say "I", not "we" in the Action — they're evaluating you. Credit the team in the Result.
  • End on a lesson or systemic fix, not just an outcome. Seniors generalize from incidents.
  • Keep to ~90 sec. Stop and let them probe. Long monologues read as junior.
  • Начни с масштаба/ставок в S, чтобы интервьюер понял, что проблема сложная (6.2 млрд записей, 100+ консьюмеров, €1M+, 8 брендов).
  • Говори "я", не "мы" в Action — оценивают тебя. Заслугу команды отдай в Result.
  • Заканчивай на уроке или системном фиксе, не просто на результате. Senior-ы обобщают из инцидентов.
  • Укладывайся в ~90 сек. Останавливайся и дай им добить. Длинные монологи читаются как junior.

Mentorship & raising standards (the senior differentiator)Менторство и повышение стандартов (отличие Senior-а)

How to set standards on a team that has noneКак установить стандарты в команде, где их нет

1. Diagnose the PAIN → defects in prod, dup logic, slow onboarding 2. Write EXEMPLARS → one model done right people can copy 3. Lightweight REVIEW → start small, blocking vs. nit, explain the WHY 4. Show the VALUE → "review caught N defects, query got Mx faster" 5. EXPAND + co-own → fold in their input → standard becomes "ours"
1. Диагностируй БОЛЬ → дефекты в проде, дубли логики, медленный онбординг 2. Напиши ОБРАЗЦЫ → одна правильная модель, которую можно копировать 3. Лёгкое РЕВЬЮ → начни с малого, блокер vs. nit, объясни ПОЧЕМУ 4. Покажи ЦЕННОСТЬ → "ревью поймало N дефектов, запрос стал в M раз быстрее" 5. РАСШИРЯЙ + совладей → вплетай их инпут → стандарт становится "нашим"
  • Review is teaching, not gatekeeping. Explain why a query plan is faster; separate blockers from nits; praise good patterns so the bar is also aspirational.
  • Review toward independence. Heavy early, deliberately lighter over time; track whether they catch their own issues.
  • Bus-factor of one is a bug. Runbooks per pipeline (purpose, owner, failure modes, remediation), version-controlled standards, exemplars.
  • Promotion = lagging indicator. 3 promotions at McMakler are the proof the bar rose for people, not just code.
  • Ревью — это обучение, не gatekeeping. Объясни почему план запроса быстрее; разделяй блокеры и nit-ы; хвали хорошие паттерны, чтобы планка была ещё и вдохновляющей.
  • Ревью к самостоятельности. Тяжёлое вначале, нарочито легче со временем; отслеживай, ловят ли они свои собственные проблемы.
  • Bus-factor один — это баг. Runbook на каждый пайплайн (цель, владелец, режимы сбоя, устранение), стандарты в системе версионирования, образцы.
  • Повышение = запаздывающий индикатор. 3 повышения в McMakler — доказательство, что планка выросла для людей, не просто для кода.

Conflict, pushback & influence without authorityКонфликт, сопротивление и влияние без власти

Recipe for "they resisted my standard / I influenced without authority"Рецепт для «они сопротивлялись моему стандарту / я влиял без власти»

  • Lead with the cost of the status quo, not authority — defects reaching prod, duplicated logic, un-reconcilable cross-brand numbers.
  • Pilot on one domain, gather data (defect-catch rate, speed-up).
  • Let proof convert skeptics; get one influential early adopter.
  • Co-own the rules — adjust the checklist with their input so it's "ours." Skeptics become reviewers.
  • Начинай с цены статус-кво, не власти — дефекты доходят до прода, дублирование логики, несводимые кросс-брендовые цифры.
  • Пилот на одном домене, собери данные (процент пойманных дефектов, ускорение).
  • Пусть доказательства конвертируют скептиков; найди одного влиятельного раннего адоптора.
  • Совладей правилами — скорректируй чеклист с их инпутом, чтобы он стал «нашим». Скептики становятся ревьюерами.
Disagree & commit "I argue hard for the right number, then I support the decision and document the tradeoff." Shows ego-strength + collaboration.
Не согласись и поддержи «Я жёстко спорю за правильную цифру, затем поддерживаю решение и документирую компромисс.» Показывает силу эго + коллаборацию.
Interviewer will probe "What if they still resisted after the data?" → escalate calmly with the risk framed in their terms (their on-call pain, their broken metric), never as a personal win.
О чём спросит интервьюер «Что, если они всё равно сопротивлялись после данных?» → эскалируй спокойно, обрамляя риск в их терминах (их боль on-call, их сломанная метрика), никогда как личную победу.
